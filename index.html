Git branching

If multiple people are working on a project, they should not work directly on the master branch. Instead, they should work on their own local branches and merge them back to the master. This way, the master is always deployable.

Create a branch

In the Git project, do: git checkout -b my_branch or: git branch -d my_branch

It's a good idea to use a descriptive name for the branch, such as the issue/ticket you're working on. You can have multiple local branches and switch between them with git checkout.

You can keep the branch local, or push it with: git push -u origin my_branch (You'll have to do this eventually to merge it)

You can work on this branch, committing/pushing changes, and then open a pull request on GitHub once you think it's ready to merge.

Pull request

It's usually a good idea to assign the request to someone other than yourself for peer review. The person reviewing the request can then review and close or approve it. Merging can usually be done automatically, and then the source branch can be deleted.

Delete branch

This should be done both on GitHub and locally. To do it locally: git pull # The master should now have all the changes git branch -d my_branch # This will not do anything is the branch hasn't been merged. Use -D to force-delete it.

Things that should not be in Git repos

  • Passwords. 'Nuff said.
  • Command histories from interactive shells (.Rhistory, python_history, etc.). Command histories are user-specific, and good code should be user-agnostic.
  • Large binary files. The amount of data stored locally is multiplied per user per branch.
    • You may have to store some things, though, like images.

.gitignore

This sits in the repo's root directory and tells Git what files to ignore when detecting changes to commit. Syntax uses Linux-style paths and wildcards. Use a trailing slash to denote a directory. For example: .python_history pycache/ this/that/other.txt one/two//.sh

Merge conflicts

If two branches diverge (e.g. someone else merges their branch to the master before you), you may have to do some fiddly manual merging. To help avoid this, you can merge changes to the master back into your branch: git pull origin master
